# Rate limits

InboundSage applies per-tenant rate limits to protect the platform and comply with Meta throughput rules.

## Response headers

Every API response may include:

| Header | Description |
| ------ | ----------- |
| `X-RateLimit-Limit` | Max requests per window |
| `X-RateLimit-Remaining` | Requests left in current window |
| `X-RateLimit-Reset` | Unix timestamp when the window resets |

Default limit is **100 requests per 60 seconds** per tenant (configurable server-side).

## Exceeded limit

When limited, the API returns `429` with:

```json
{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "Too many requests",
    "docs": "https://docs.inboundsage.com/errors/RATE_LIMIT_EXCEEDED"
  }
}
```

## Campaign sending limits

Campaign workers respect **per phone number** rate limits and Meta tier caps via an internal token bucket. Campaigns may auto-pause with status `PAUSED` when daily limits or WABA quality (`RED`) block further sends.

## Idempotency

Safe retry for `POST` operations without duplicate side effects:

| Endpoint | Header |
| -------- | ------ |
| `POST /v1/messages/send` | `Idempotency-Key` or `X-Idempotency-Key` |
| `POST /v1/campaigns` | `Idempotency-Key` or `X-Idempotency-Key` |

### Behavior

- Same tenant + key within **24 hours** returns the cached response (including status code).
- Same key with a **different body** returns `409` `IDEMPOTENCY_CONFLICT`.
- Omitting the header allows duplicate requests (not recommended for production).

### Example

```bash
curl -s -X POST https://api.inboundsage.com/v1/campaigns \
  -H "Authorization: Bearer $TOKEN" \
  -H "Idempotency-Key: create-campaign-v1" \
  -H "Content-Type: application/json" \
  -d '{ ... }'
```

Replay the same request with the same key to get the original `201` response without creating a second campaign.
