# Authentication

## API keys

All API requests require an API key passed in the `Authorization` header:

```
Authorization: Bearer is_live_YOUR_KEY
```

Keys are prefixed with `is_live_` (production) or `is_test_` (test mode).

<Note>
  Create and manage API keys in the
  [dashboard](https://app.inboundsage.com/api-keys).
  Keys are shown only once at creation time.
</Note>

Each key is scoped to one **tenant** (workspace) and grants access to that workspace's WABAs, templates, campaigns, and webhooks.

```bash
curl -s https://api.inboundsage.com/v1/wabas \
  -H "Authorization: Bearer is_live_YOUR_KEY"
```

<Warning>
  Never commit API keys to source control or expose them in browser-side code.
</Warning>

## JWT (dashboard sessions)

For tools built on top of the logged-in dashboard, authenticate users with JWT:

```bash
curl -s -X POST https://api.inboundsage.com/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"you@company.com","password":"your-password"}'
```

Use the returned `accessToken` as the Bearer token. Refresh via `POST /v1/auth/refresh`.

Developer endpoints (`/v1/campaigns`, `/v1/webhooks`) require **OWNER** or **ADMIN** when using JWT. API keys are not role-restricted but remain tenant-scoped.

## Rate limits

Every response includes rate limit headers:

| Header | Description |
|--------|-------------|
| `X-RateLimit-Limit` | Requests allowed per window |
| `X-RateLimit-Remaining` | Requests remaining in current window |
| `X-RateLimit-Reset` | Unix timestamp when the window resets |

When you exceed the limit, the API returns `429` with error code `RATE_LIMIT_EXCEEDED`.

Default limit is **100 requests per 60 seconds** per tenant. See [Rate limits](/rate-limits) for campaign throughput and Meta tier behavior.

## Idempotency

For `POST` requests, pass an `Idempotency-Key` header to safely retry without duplicate operations:

| Endpoint | Header |
| -------- | ------ |
| `POST /v1/messages/send` | `Idempotency-Key` or `X-Idempotency-Key` |
| `POST /v1/campaigns` | `Idempotency-Key` or `X-Idempotency-Key` |

- The same key returns the original response for **24 hours** (per tenant).
- Same key with a **different body** returns `409` with `IDEMPOTENCY_CONFLICT`.
- Omitting the header allows duplicate requests (not recommended in production).

```bash
curl -s -X POST https://api.inboundsage.com/v1/campaigns \
  -H "Authorization: Bearer is_live_YOUR_KEY" \
  -H "Idempotency-Key: create-campaign-v1" \
  -H "Content-Type: application/json" \
  -d '{ ... }'
```

## Multi-tenancy

Tokens are bound to a single tenant. Webhooks, campaigns, and inbound events only match resources in that workspace — create keys and webhooks under the same tenant as your connected WABA.
