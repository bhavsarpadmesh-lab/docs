# Authentication

## API keys

All API requests require a Bearer token in the `Authorization` header:

```
Authorization: Bearer is_live_YOUR_KEY
```

### Key prefixes

| Prefix | Type | Use |
|--------|------|-----|
| `is_live_` | Production API key | Server-side REST calls |
| `is_test_` | Test API key | Sandbox / integration tests |
| `is_whsec_` | Webhook signing secret | Verify `X-InboundSage-Signature` only — not an API credential |

<Note>
  Create API keys in the [dashboard](https://app.inboundsage.com/api-keys).
  Keys are shown **once** at creation. Webhook secrets are returned once when you `POST /v1/webhooks`.
</Note>

Each API key is scoped to one **tenant** (workspace) and grants access to that tenant's WABAs, templates, campaigns, and webhooks.

```bash
curl -s https://api.inboundsage.com/v1/wabas \
  -H "Authorization: Bearer is_live_YOUR_KEY"
```

<Warning>
  Never commit API keys or webhook secrets to source control or expose them in browser-side code.
</Warning>

## JWT (dashboard sessions)

For apps built on the logged-in dashboard, obtain a JWT:

```bash
curl -s -X POST https://api.inboundsage.com/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"you@company.com","password":"your-password"}'
```

Use `accessToken` as the Bearer token. Refresh with `POST /v1/auth/refresh`.

Developer endpoints (`/v1/campaigns`, `/v1/webhooks`) require **OWNER** or **ADMIN** when using JWT. API keys are not role-restricted but remain tenant-scoped.

## Rate limits

Limits apply per API key / tenant. Plan tiers:

| Plan | Requests/minute |
|------|----------------|
| Starter | 100 |
| Growth | 500 |
| Scale | 1000 |

Every response includes:

| Header | Description |
|--------|-------------|
| `X-RateLimit-Limit` | Requests allowed per window |
| `X-RateLimit-Remaining` | Requests remaining in the current window |
| `X-RateLimit-Reset` | Unix timestamp when the window resets |

When exceeded, the API returns `429`:

```json
{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "Too many requests. Retry after 1716595200.",
    "docs": "https://docs.inboundsage.com/errors/RATE_LIMIT_EXCEEDED"
  }
}
```

Use the `Retry after` timestamp (also `X-RateLimit-Reset`) before retrying. See [Rate limits](/rate-limits) for campaign throughput and Meta tier behavior.

## Idempotency

Pass `Idempotency-Key` or `X-Idempotency-Key` on supported `POST` endpoints to safely retry network failures without duplicate sends or campaigns.

| Endpoint | Idempotent |
|----------|------------|
| [`POST /v1/messages/send`](/api-reference/messages/send) | Yes |
| [`POST /v1/campaigns`](/api-reference/campaigns/create) | Yes |

| Rule | Behavior |
|------|----------|
| Same key, same request (within **24h**, per tenant) | Returns the cached response (HTTP status + body) |
| Same key, different body | `409` — `IDEMPOTENCY_CONFLICT` |
| Key longer than 255 chars | `400` — `VALIDATION_ERROR` |
| Header omitted | Request is always processed (not recommended in production) |

### Message send

```bash
curl -s -X POST https://api.inboundsage.com/v1/messages/send \
  -H "Authorization: Bearer is_live_YOUR_KEY" \
  -H "Idempotency-Key: send-msg-001" \
  -H "Content-Type: application/json" \
  -d '{
    "waba_id": "YOUR_WABA_ID",
    "phone_number_id": "YOUR_PHONE_NUMBER_ID",
    "to": "919876543210",
    "type": "template",
    "template": { "name": "your_template", "language": "en" }
  }'
```

### Create campaign

```bash
curl -s -X POST https://api.inboundsage.com/v1/campaigns \
  -H "Authorization: Bearer is_live_YOUR_KEY" \
  -H "Idempotency-Key: campaign-launch-001" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "May Offer",
    "wabaId": "YOUR_WABA_ID",
    "phoneNumberId": "YOUR_PHONE_NUMBER_ID",
    "templateId": "YOUR_TEMPLATE_ID",
    "audienceType": "INLINE",
    "contacts": [{ "phone": "919876543210" }]
  }'
```

Use a unique key per logical operation (e.g. `send-order-48291`, not a static string reused across different recipients).

## Multi-tenancy

Tokens are bound to a single tenant. Webhooks, campaigns, and events only apply to resources in that workspace — use API keys and webhooks under the same tenant as your connected WABA.
